Horizon UI · the catch "trusted" wasn't safe

the cleanest recall win

A trusted Chakra search box, with no label.

This is Chakra's own Input — a trusted, well-built component — used as the site search. It has a placeholder, "Search…", and nothing else. No label, no aria-label, no id tying it to one. A screen-reader user lands on it and hears nothing about what it's for.

Chakra guarantees the input renders correctly. It can't guarantee you gave it a name. The content check did what the library can't.

The pattern trusted ≠ safe

the same gap, two libraries

Both projects hid a real bug behind a "trusted" component.

An Ant Design button with no name. A Chakra input with no label. Two different libraries, the same kind of bug — and a structural linter waves both of them through, because the component comes from a library it trusts. The library got its own internals right. It said nothing about the content the app passed in.

"Trusted" guarantees the structure. The content check guarantees the name. You need both.

The honest read low coverage is partly correct

the number that doesn't matter

"Only 9 elements checked" is the wrong thing to measure.

A fair pushback: the checker directly inspected only nine or ten elements per project. Doesn't that mean it's mostly blind? No — and this is the honest part. On a well-built design-system app, most components should be trusted, because the library really does guarantee its own primitives. Counting those as "unscanned" would be the lie.

The number that matters isn't coverage percentage. It's did it find the real, app-level bugs without crying wolf. On both projects, cold, the answer was yes.
