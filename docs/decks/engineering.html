The problem why a linter isn't enough

the blind spot

A linter only understands real HTML. Your code isn't real HTML.

A normal accessibility linter knows about button, a, and img. But your codebase isn't written in raw tags — it's written in your own components. SubmitButton, NavLink, Avatar. The linter has never heard of any of them, so it quietly skips them and moves on.

That's the real trap: it stays silent about what it skipped. Your report comes back clean, and clean reads as safe — when the tool was actually blindfolded for most of the screen. The first job of this checker is to take that blindfold off — and stay honest about what it still can't see.

App.tsx — as a plain linter sees it
<SubmitButton />
// linter: "SubmitButton"? not a tag I know.
// → skipped, silently

<NavLink to="/pricing"/>  // skipped
<Avatar src={user.img} /> // skipped
↳most of your UI, never looked at0 checked

The shape the whole system in three words

everything reduces to this

Two ways to look. One source of truth.

A Structure passIs this the right kind of element? Runs the proven rules on every element it can identify.
B Content passDid you actually give this control a name? Reaches the components the structure pass can't see.
✦ The corpusReal audit evidence. Turns "this is wrong" into "this is wrong, it's common, here's the fix."

It looks at your code two different ways, and it judges what it finds against one body of real-world evidence. Structure, then content, then weighed against what actually breaks in the wild. The rest of this is those three things, unpacked.

●03 / 12
Stage 1 · Resolve what is this component, really?

wrapper → host

First, figure out what each component really is.

For every component on the screen, the checker answers one question: underneath the wrapping, what plain element is this? It tries four ways, strongest first — you declared it, it's in our registry of known libraries, we can trace its source, or we can't tell.

That last case is the honest one. Instead of hiding what it couldn't resolve, it reports it — and splits the unknowns into buckets so the number doesn't lie.

when it can't tell — the honest buckets

trusted

From a known-good design system. The library handles its own insides.

icons

An icon. No interactive element to check. Not a gap.

structural

Plumbing — providers, layout, fragments. Nothing to check.

declare

A genuine unknown. The only real gap — and the only one we ask you to fix.

Stage 2 · Detect two passes, one example

the recall win

The bug that hides behind the word "trusted."

Here's a real button from a popular library, used with just an icon inside and no label. A screen reader announces it as "button" — nothing else. The user has no idea what it does. That's a genuine bug.

The structure pass misses it completely: the library is trusted, so the structure really is fine. But "trusted" only covers the library's insides — it says nothing about the content you passed in. The content pass is the one that catches this, and it's where most of the value lives.

Toolbar.tsx
// from a trusted design system
<IconButton>
  <TrashIcon />
</IconButton>
↳icon-only · no accessible nameWCAG 4.1.2

Structure pass: missed it. Content pass: caught it.

●05 / 12
The discipline uncertain → skip

the most important rule in the system

It only speaks when it can prove the problem.

The content pass never asks "does this have a name?" It asks "can I prove it doesn't?" The moment anything is uncertain, it goes quiet. It would rather miss a real problem than invent a fake one — because a tool that cries wolf gets switched off, and a switched-off checker catches nothing.

  • <IconButton><TrashIcon/></IconButton> icon-only, no name — certain ✦ FIRES
  • <IconButton aria-label="Delete"> it has a name ○ SKIP
  • <IconButton {...props}> props could carry a name — unknowable ○ SKIP
  • <IconButton aria-label={x}> dynamic value — can't read it ○ SKIP
  • <Tooltip title="Delete">…</Tooltip> the tooltip names it at runtime ○ SKIP

Stage 3 · Enrich how much should you care?

finding → priority

Every finding carries how much it matters.

A raw finding is just "this is broken." That's not enough to act on — a list of two hundred equally-shouting problems gets ignored. So each finding is matched to the corpus by its criterion and picks up two things: how widespread that failure is across real audits, and the fix that actually worked.

Now it reads like a priority, not a warning.

enriched finding
finding  no accessible name
wcag     4.1.2
tier     VERY COMMON · 21/26 orgs
fix      give the control an aria-label,
         or visible text
↳now it sorts to the top of the listpriority
●07 / 12
Stage 4 · Contract binclusive.json + the AGENTS block

one committed file

A file that teaches the tools your rules.

Each repo gets a small binclusive.json, committed with the code. It does three jobs: an escape hatch for what the checker can't infer, a policy for which failures block a build versus only warn, and the rules your team has learned.

From that file it writes a block into your AGENTS.md and CLAUDE.md. That's the quiet payload — it puts the corpus rules in front of the AI before it writes a line, so the right thing is the default, not the correction.

AGENTS.md — generated block
<!-- BEGIN binclusive (generated) -->
## Accessibility contract
block: 1.3.1, 4.1.2, 2.4.4
warn:  1.1.1, 2.1.1

most common first:
- 1.3.1 · 22/26 orgs · label every field
- 4.1.2 · 21/26 orgs · name every control
<!-- END binclusive -->
↳the AI reads this firstdefault, not fix
●08 / 12
Stage 5 · Distill the determinism boundary

how the corpus gets built

Judgment offline, frozen as data. Everything that ships is deterministic.

The one genuinely hard step is grouping messy findings — written by different auditors, in different words — into the same underlying failure. A regex is bad at that; a language model is good at it. So we run the model once, offline, and freeze its judgment into a reviewable file — like a fixture.

LLM · offline · once

Reads anonymized findings, groups them into failure-shapes, writes the generic English prose. Committed and human-reviewed. Then it stops.

code · every run · replayable

The privacy gate (a pattern needs 3+ companies), the counting, the anonymizing, the ledger. No model in the loop — nothing unpredictable ever ships.

The north star how the next tool should work too

if we rebuilt it tomorrow, we'd keep these

01ConservatismSpeak only when you can prove it. Silence beats a false alarm.
02HonestyShow what you couldn't check. A fake-clean report is worse than an admitted gap.
03Parse, don't validateHandle messy data once, at the edge, and fail loud — never smuggle "unknown" inward.
04One-way generationGenerated things come from a source and are never hand-edited. Guard the drift in CI.
05Determinism boundaryKeep judgment offline and frozen. Ship only code you can replay.

The loop why it compounds

step back

Every audit makes the checker sharper.

audits find failures → corpus → checker catches them at write-time → cleaner code ships ↺

The tool isn't a snapshot — it compounds. It gets better every time Binclusive runs an audit, and no generic tool can copy that, because no generic tool has the audits.

●11 / 12
Where it goes honest edges + what's next

no fake-clean endings

Honest edges, and what's next.

Part of the corpus is still a hand-built seed; the full distillation is rolling out criterion by criterion. The content pass covers the highest-value control types today — buttons, links, images, inputs, dialogs — and that list will grow. Every new failure shape we learn to recognize is one more thing the AI gets right before we ever have to ask.

That's the whole machine. Questions are the fun part.
