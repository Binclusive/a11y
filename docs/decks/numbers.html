The question what this deck proves

measured, not asserted

Two claims have to be true. So we measured both.

Two things have to hold for any of this to matter. One: the audit corpus has to be a real moat, not a nice idea. Two: the checker has to work on code we didn't write — including code that looks nothing like our own.

This deck is the measurement of both. It keeps the parts where we got it wrong, because those are the parts that make the rest believable.

Act 2 · Why it's the moat the same head

the kicker

Our failures are the AI's failures.

Automated tools are great at what a machine can measure cheaply — contrast, missing alt text. Our corpus is built from the other half: the semantic and interaction failures that take a human to spot. Structure, names, keyboard, focus.

And here's the kicker — that's the exact half AI coding tools get wrong. They reach for a clickable div, they forget focus, they skip live regions. Our audit head and the AI's failure head are the same head. So an off-the-shelf checker handles the floor, and our corpus handles the tail — and our tail is pre-aimed at the thing writing the code.

Act 3 · The honest part overfit → fixed

no fake-clean

First, it didn't generalize.

The first real run looked great — on the one repo we'd tuned against. Zero false positives. Then we pointed it at three fresh projects it had never seen, and it broke. One came back at forty percent false positives.

The lesson was blunt: harden on one repo and you overfit to that repo. Only fresh, different code tells the truth. So we went back and fixed the causes, not the symptoms.

The biggest false-positive source had one root cause: the checker recognized a control by its name alone, and different libraries reuse the same names for different things. ●11 / 13
No fake-clean the easiest place to lie is a clean number

said out loud, on purpose

What the numbers don't say.

A clean-looking number is the easiest place to lie, so here are the edges. Coverage percentage is a vanity metric here — on a well-built design-system app, low coverage is partly correct, because the library really does guarantee its own primitives. The honest metric is "did it find the app-level bugs with almost no false positives," and that's the one we report.

Part of the corpus is still a hand-built seed while the full distillation rolls out. And dialogs are genuinely hard to judge from static code, so we hand those to the live audit. None of that hides in a headline number — which is exactly why we say it out loud.
