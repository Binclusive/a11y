Everything is one pipeline. We work out what each component really is, look for problems two different ways, weigh what we find against real audit data, let each repo set its own policy, and — offline — build the corpus that powers all of it. The rest of this deck is the hard problems inside those five moves, and what we did about each.

Problem 1 · Resolve the foundational move

where everything starts

A linter only sees real HTML tags. Your code is components.

PROBLEM
Linters check button, a, img. Your code is SubmitButton, NavLink, Avatar — skipped, silently.
APPROACH
Teach the linter what each component really is — a map from every wrapper to the element it renders.
SOLUTION
Source-tracing: follow the import, read the syntax tree, and if it forwards its props to exactly one element, record that element as the host.

four ways, strongest first

Resolve four ways — and the real code that fought back.

Real open-source code is where this got hard. A monorepo hides its button in an internal package; an import path is an alias, not a folder; shadcn's asChild swaps the element through a Slot; a barrel file re-exports from a sibling. Each broke tracing once — and each got a real fix, because if we can't follow a workspace design system, we can't see anything.

Resolve · honesty a silent skip looks like clean code

honesty as a feature

When we can't resolve it, we say so.

The easy mistake is to skip what you can't resolve and report a clean scan. That's a lie — it makes a design-system app look 94% fine when you were blind to most of it. So we report the unknowns, and split them honestly. Three of these aren't gaps. Only declare is a real to-do.

trusted

known-good library — opaque but fine.

icons

no interactive element to check.

structural

plumbing — providers, layout, fragments.

declare

a genuine unknown — the only real gap.

Problem 2 · Detect the recall win

the second pass

"Trusted" guarantees the structure — not the content you passed in.

PROBLEM
A trusted <IconButton> with just an icon inside is a real nameless-control bug. The structural pass calls it fine — the library is trusted.
APPROACH
Check the content the app owns — name, alt, label — at the call site, without needing to see inside the component.
SOLUTION
A second pass that recognizes the control type at the call site and verifies the app-owned content — so it reaches the opaque, trusted components the first pass can't.
●06 / 15
Detect · the discipline the most important rule we set

the bias that makes it shippable

We never ask "does it have a name?" We ask "can we prove it doesn't?"

A content check has one way to fail badly: false positives. Yell about a name that's actually there — somewhere you couldn't see — and engineers turn the whole thing off. So we made one rule absolute: the second pass speaks only when the problem is statically, provably there. Spread props, a value built at runtime, a label in a sibling — any uncertainty, and it goes quiet.

A tool that cries wolf gets switched off, and a switched-off checker catches nothing.

●07 / 15
Detect · the guards conservatism, made concrete

every one is a real false positive we hit and closed

The specific guards that keep it quiet.

  • ·strength lever — recognized by a proven host fires harder than recognized by name alone.
  • ·spread & dynamic — {...props} or a computed value could carry the name → skip.
  • ·toggles — a checkbox's name is the row text beside it, which we can't see → skip.
  • ·label ancestors — an input under a <label> or form-field is named by it.
  • ·name-injecting wrappers — a titled <Tooltip> names its child at runtime.
●08 / 15
Case study · root cause the deepest fix

names stopped counting as proof

22 false positives → 1. The fix was about evidence.

On react-admin the content pass fired 22 times, almost all wrong — from two separate roots. It recognized a control by name alone, and react-admin names a button with a label prop and no children, so every Button looked nameless. And the linter's component map is keyed by name only, so a Material UI TextField made react-admin's display TextField — a span — read as an input everywhere.

The fix for both: demand module-scoped evidence. A component is that control only if it resolved to a real host from that exact module. A shared name proves nothing.

react-admin — enforce findings
22→1
21 false positives gone. The 1 residual is a documented dialog case, kept on purpose.

Problem 3 · Distill the determinism boundary

building the corpus

Real audit data is messy, bilingual, and won't group by rule.

PROBLEM
643 findings, two languages, many authors, selectors that are guesses. Grouping "the same failure" is the one step code is bad at.
APPROACH
Use a language model for exactly that one step — and nothing else in the shipped pipeline.
SOLUTION
Run the model once, offline, and freeze its grouping into a reviewable file. Everything downstream is deterministic code you can replay.
●10 / 15
Distill · deterministic half everything after the freeze is replayable

plain code, end to end

After the freeze, it's all replayable.

We normalize the messy criterion field into a real code, look up each finding's group from the frozen file, and keep a pattern only if at least three different companies hit it — the privacy gate and the "is this real" gate in one. Then we strip every identifier, collapse free-text journeys into a fixed set of categories, and count everything we dropped. Nothing vanishes silently.

1normalize"wcag244", "WCAG 2.4.4", axe rule-ids → one SC code.
2grouplook up the finding's frozen cluster (the model's offline call).
3gatekeep it only if 3+ different companies hit it.
4anonymizestrip org ids; journeys → a closed enum.
5ledgercount everything dropped — no silent losses.

Robustness three habits that kill bug classes

not handled — impossible

Make bad states impossible, not handled.

AParse, don't validateRequired fields fail loud at the boundary; the optional config degrades one bad line at a time — punishing the escape hatch is how people stop using it.
BImpossible states unrepresentableModel resolution as a discriminated union, so a contradiction like "opaque, but has a host" literally can't be written.
COne-way generationThe rules the AI reads are generated from one source, never hand-edited. A CI check fails the build on drift.
●12 / 15
Delivery a finding in a report is too late

the fix arrives with the code

We put the corpus in front of the AI before it writes a line.

›the AGENTS blockgenerated from the corpus into AGENTS.md / CLAUDE.md — the AI reads the most common failures first.
›the hookthe instant the AI edits a file, the checker runs on it and whispers any findings back into the same turn.
›the MCP serverthe same checks exposed to Cursor, Copilot, and Claude, on demand.

A finding after the fact is the slow loop we're killing. So the corpus reaches the model three ways, and the fix arrives in the same breath as the code.

Validation clean on our own repo means nothing

only different code tells the truth

We proved it on code we didn't write — and kept the misses.

9
real public OSS projects — different frameworks and design systems
214
tests green · no false negatives
0
unexplained false positives across the 5-repo catalog

Our own clean repo gave false confidence, so we ran it on nine real projects. It overfit at first — great on the repo we tuned on, 40% false positives on a fresh one. That's the honest test. We fixed the root causes, not the symptoms, and the catalog came back with every false positive either fixed or written down.
